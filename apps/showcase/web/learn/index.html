Shrimp — how it works

The bot you’re playing is Shrimp, an AlphaZero-family agent. One neural network reads a position and returns a policy (a distribution over legal moves) and a value (who is winning); a Gumbel Monte-Carlo tree search uses that network to look ahead and returns a move stronger than the raw policy; and the network is trained entirely on its own games to imitate what the search found. The loop uses no human game database; an optional behavioral-cloning warm start can precede it. This page is an overview, and the remaining pages describe each part.

01the loop

Everything below runs as one repeating cycle, one epoch at a time. Self-play generates games with the current network; the trainer samples recent games and nudges the network toward the search’s conclusions; the improved network is checkpointed and served back to self-play. Evaluation runs outside the loop and its verdict is informational: it measures strength against a pool of frozen opponents and does not gate or roll back a checkpoint.

Search budgets differ by role: self-play full searches use 1024 visits, self-play fast moves 192, and the evaluation arena 512. A stronger net needs fewer visits for equal move quality, so these budgets are recalibrated as the run matures.

03properties of the design

Variable-size input. The board is unbounded and a game can extend anywhere on the grid, so a fixed-size input plane is not usable. Shrimp builds a support set (stones, legal moves, and a one-cell halo) and feeds the network a variable-size graph that grows and shrinks with the game. Cells are ordered legal-first, which gives the legal-prefix property: the policy head’s first legal_count outputs are exactly the legal moves, so the network cannot score an illegal one. How the support set is built →

Global attention over an irregular point set. Convolutions on this graph are direction-typed: the neighbour along each of the six hex axes gets its own weights, because the axis a line runs along determines its tactical meaning. A convolution spreads information one ring per block. Interleaved attention blocks give every cell a view of the whole position in a single step: all-pairs attention over the support cells plus 8 learned summary tokens, with a relative-position bias keyed on the exact hex offset between two cells (a 217-offset disk, then ring buckets). A threat forming on the far side of the board is visible after one attention block. How attention handles hex geometry →

Gumbel-Top-m root selection. The search draws m = 32 root candidates by adding Gumbel noise to the policy logits, then runs Sequential Halving so the visit budget concentrates on survivors. Sampling without replacement gives exploration coverage at the root, and the played move improves on the raw policy in expectation even at small budgets. The improved root distribution is the policy training target (policy_target = "gumbel"), and threat-space search force-includes tactical cells so the candidate draw cannot miss a win-in-one.
