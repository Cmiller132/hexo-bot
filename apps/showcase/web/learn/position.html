The position — what the network reads

The network never sees a padded grid. Every position is rebuilt, from raw placement facts, into a support set — exactly the cells that matter right now — and every cell in it carries 46 feature planes plus 12 blocker-aware ray lengths. This page is what those numbers are: the three-layer support build, the complete plane map, and live versions of the window scan and ray walk that produce most of the planes. Every figure below runs the same geometry as the production featurizer.

01the support set

Three layers, rebuilt per position:

  • Stones — every occupied cell.
  • Legal cells — every empty cell within hex-distance 4 of a stone (the support radius; the rules themselves allow distance 8).
  • Halo — the one-cell shell just outside, so a convolution centered on an edge cell still has neighbours to read.

One multi-source BFS outward from the stones produces all three layers plus each cell’s distance-to-nearest-stone in a single pass. Early positions have a few dozen nodes, crowded ones a few hundred.

Nodes are stored in a fixed order — legal cells, then stones, then halo, each segment sorted by packed coordinate id (ascending signed (q, r)). Legal-first gives the legal-prefix property: the policy head emits one logit per node in node order, so the first legal_count outputs are the legal moves. Nothing downstream ever matches coordinates against a rules oracle — taking the first legal_count logits is the mask, a slice that cannot drift out of sync with the node order.

To the network a position is five aligned arrays over the N nodes: feats (N × 46), coords (N × 2) axial coordinates, nbr (N × 6) neighbour row-indices along the six hex directions (a missing neighbour is a sentinel that reads as a zero row), raylen (N × 12) ray lengths (§04 below), and a boolean node mask used only for batch padding.

Real scale: the ply-40 self-play position traced on the end-to-end page builds N = 402 nodes = 288 legal + 40 stones + 74 halo — so its policy output is 402 logits of which the first 288 are exactly the legal moves.

02forty-six planes per cell

The featurizer is implemented twice — Rust for production, Python as the reference oracle — kept identical by exact-parity tests. Stone and threat planes are side-to-move relative: “own” always means the player about to move, whichever colour that is. One colour plane (index 6) carries the absolute side, and it is the only one that does.

The 46 planes split into 11 base scalars, 30 axis-indexed window planes, 2 fork scalars, and 3 global scalars. Under a lattice symmetry the 16 scalar planes stay in place while each window quantity’s {Q, R, QR} triple permutes — the typed layout the symmetry machinery acts on.

Each window quantity’s three axis values are contiguous (plane = 11 + quantity × 3 + axis), so a 60° rotation acts on the 30 axis planes as ten independent 3-cycles and a reflection as ten transpositions. The three globals — ply, distance to the stone centroid, and spread — are all symmetry-invariant; on an empty board ply and dist-to-centroid stay 0 and the spread plane reads 1/16.

03the window scan

A window is one of the 6 possible length-6 alignments through a cell along one win axis (Q, R, or QR) — 18 windows per cell in all. A window is clean for a side if it contains no enemy stones: it could still become that side’s six-in-a-row. For every support cell and every axis the featurizer scans all 6 windows and emits:

  • own_line, opp_line — the maximum stone count among that side’s clean windows, /5: how advanced the best line through this cell is. (A clean window holds at most 5 own stones in a decision state — 6 is a played win.)
  • own_live, opp_live — the count of clean windows, /6: how open this cell is on this axis.
  • own_live3/4/5, opp_live3/4/5 — the count of clean windows already holding ≥ 3, ≥ 4, ≥ 5 side stones, /6. Graded threat multiplicity: live5 is per-cell standing-win multiplicity, live4 counts distinct win-in-two threats.

Two fork scalars follow the 30 axis planes: the number of axes (of 3) whose raw line count reaches 3, per side, /3 — a cell that forks is a cell whose best lines on two or three different axes are all advanced at once.

Windows that run off the support count as clean-and-empty: an absent cell contributes no stones. Edge cells therefore read slightly more open than they are. The convention is deliberate — it is player-symmetric and identical in both featurizers, so neither side and neither implementation ever disagrees about what an off-support cell means.

The colours are side-relative, so flipping the side to move swaps them: a window clean for blue when blue moves is the same window clean for “opp” when red moves. Filter to one axis and place a stone of each colour on it — the moment a window holds both colours it goes dead for both sides, and the live counts drop together.

04the ray lengths

Separately from the planes, every node carries 12 ray lengths: for each side × axis × direction, how far a straight walk from the cell stays attendable. The walk rules:

  • a cell off the support stops the walk;
  • an enemy stone is included — the blocker itself is visible — and then stops it;
  • own stones and empty cells pass through — a side’s line of sight is never blocked by its own stones.

Values run 0–5. The reach of 5 is the window-6 geometry made exact — a length-6 window through a cell extends at most 5 cells along its axis — not a tunable knob. The flat order is side × 6 + axis × 2 + dir with side own-first, axes [Q, R, QR], and dir {+, −}.

Ray lengths are not input planes. They feed the trunk convolutions directly: each conv’s six direction taps read a visibility-masked, distance-weighted span of up to 5 cells, masked by exactly these lengths — own-side visibility on half of each orbit’s channels, opp-side on the other half. That mechanism lives on the network page; here you can see the raw quantity it consumes.

The two sides’ rays differ only in who counts as the enemy: an own-side ray walks through own stones and stops one cell into a red stone (when blue is to move); the opp-side ray from the same cell walks through red stones and stops at blue ones. Flip the visibility toggle on a cell next to stones of both colours and watch the same six directions truncate differently.

That is the network’s entire view of a position: N nodes in [legal | stones | halo] order, 46 planes and 12 ray lengths each. Next: why rotating that input rotates the output, exactly.
