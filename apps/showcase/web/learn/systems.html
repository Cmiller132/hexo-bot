The systems — the engineering under the model

The system described on the other pages runs on four pieces of infrastructure: a Rust rules engine that updates win/threat state incrementally, a game record that costs about four bytes a move, a serving path that keeps each game’s search tree alive in one process, and a ∼$100 GPU that serves the site. This page covers the non-ML engineering, read from the code.

Ground truth. Checked against packages/hexo_engine/rust/src/ (board, windows, packed action ids), packages/hexo_utils/rust/src/records.rs (the .hxr codec), packages/shrimp/rust/src/ (search sessions, eval cache), and apps/showcase/server/showcase/ plus apps/showcase/Dockerfile.xpu (the production deployment).

01the rules engine

The engine is a Rust crate behind a PyO3 bridge (hexo_engine._rust): Python, the search, and this website all hold opaque Rust state handles, and every placement goes through the same authority. An illegal move raises an error. Because the board is unbounded, storage is sparse: a hash map of occupied cells plus an insertion-ordered list, with no grid.

A stone sits inside exactly 18 six-cell windows (the three line axes of the hex grid, times the six positions the stone can occupy inside a length-6 window), and those 18 are the only windows whose status that stone can change. The engine keeps a window store: for every window ever touched, two 6-bit masks (one per colour). A placement ORs one bit into 18 masks and reads the result: a window with all six bits of one colour is a win; a single-colour window with four or more stones is a threat. The win/threat detector is constant work per move, whether the game has 10 stones or 300, with no board rescan.

Two more stores are updated the same way. The legal-move store maintains the radius-8 disk around the stones (every placement opens up to 217 new cells and closes its own), so listing the legal moves is a copy. A live-threats index mirrors the window store, so checking whether any live threat exists is an O(1) read. Threat-space search performs that check before every move. When the search force-includes a double-threat cell, these masks are the data it read. Every mutation also returns an undo delta; the search rolls positions back rather than copying them.

Moves are exchanged as packed action ids. A coordinate (q, r) becomes one u32: each axis is offset into an unsigned 16-bit lane, then q << 16 | r, so plain integer order equals deterministic (q, r) order. The same encoding is implemented byte-for-byte compatibly in Rust, Python, and the board JS on this site, because these ids are persisted in training shards and game records; a cross-language test pins them together.

02the game record

Every finished game (a self-play game on the training box, an arena game in the evaluator, or a game played on this site) is written in one binary format: .hxr. A file starts with the magic HEXOREC1 and a small header (schema version, engine rules version, backend string, the player list), then zero or more length-prefixed game payloads. Each payload is self-contained: game id, optional seed (zigzag varint), a status byte, the move stream as raw u32 little-endian packed action ids, then optional winner, placement count, and an abort record for games that died mid-run. Writers flush per game, so a crashed self-play run leaves a readable prefix rather than a corrupt file; readers reject unknown schema versions and trailing bytes.

The move stream dominates the cost: 4 bytes per ply, plus roughly 60 bytes of per-game metadata. A typical ~60-ply game is about 300 bytes. A record stores only the action stream; a consumer replays it through the engine to reconstruct every position. This site works the same way: a finished game is finalized as a one-game .hxr blob into the games database, and the analysis view replays those action ids through the same Rust engine before the network sees the position.

03the serving path

A move on this site is a POST /api/game/{id}/move to a FastAPI process that holds no model. The process checks the session cookie and a token bucket, applies the stone through the Rust engine, and enqueues a bot-turn job. Jobs route sticky: game_key % workers pins each game to one of two worker processes, the one holding that game’s search state. Each worker holds the full checkpoint catalogue, with one network and one Rust search session per checkpoint, resident for its lifetime.

Inside the worker, the session searches with the run’s as-trained settings (Gumbel root, sequential halving, TSS); only the selected visit budget varies. Two reuse layers reduce the cost. The per-game tree: the session keeps one search tree per game_key; after it plays a move it advances the root into the chosen child, so the next search usually starts from an already-grown subtree. The eval cache: every evaluated position is keyed by a state hash and kept in a FIFO-bounded map (the crate default holds 262,144 states; this site configures 65,536 per checkpoint session), so transpositions and re-searches do not hit the GPU twice. Cache misses are batched: the search collects up to 32 leaves and makes one network forward per batch. The bot plays its whole turn (one or two stones) in a single job, then the tree for a finished game is explicitly discarded on its sticky worker.

The rate limits protect search compute. Moves are capped at 120/minute and new games at 60/hour per client, at most 16 games run concurrently server-wide and 4 per IP, and analysis requests (which can trigger small searches, capped at 64 visits) get 40/minute. Identity for all of this is the Cloudflare-reported client IP; the container has no directly exposed ingress.

04the gpu it runs on

The whole site is served by one Intel Arc A310 (a roughly $100 low-profile card) through torch 2.12.1+xpu, the PyTorch wheel line where Intel-GPU support is native (torch.xpu, no extension package). The training run’s Triton kernels are CUDA-only and gated on x.is_cuda, so on this device every forward takes the plain eager fp32 paths: the same math without the hand-tuned kernels. The measured result on this deployment is 2.2× single-move speed and 2.8× concurrent throughput over the host’s 7-core CPU baseline. Batching favours the GPU, and the serving load is batched.

Eager-on-XPU is not covered by CI, so at startup each worker forwards one fixed position on CPU and on the XPU and compares value and policy logits. Tolerance is 10−2; the measured divergence here is about 5×10−6. If the check fails (mismatch, non-finite output, or the device forward raising), the worker falls back to CPU and serves slower moves.

The first forwards after a worker starts compile the SYCL kernels for the network: about 27 seconds, once per worker (two workers compile independently). A startup warmup absorbs this, so no player’s first move pays it. The container runs a read-only filesystem with tunnel-only ingress, so the kernel cache lives on tmpfs: a rebooted container pays the 27 seconds once, then serves warm indefinitely.

The supporting engineering is small: incremental window masks, four bytes per move, sticky worker processes, and a parity check before a backend is trusted.
