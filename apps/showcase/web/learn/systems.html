The systems — the engineering under the model

Everything the other chapters describe runs on a small stack of deliberately boring machinery: a Rust rules engine that never rescans the board, a game record that costs about four bytes a move, a serving path that keeps each game’s search tree alive in one process, and a ∼$100 GPU that serves this entire site. This page is the non-ML engineering, read straight from the code.

Ground truth. Checked against packages/hexo_engine/rust/src/ (board, windows, packed action ids), packages/hexo_utils/rust/src/records.rs (the .hxr codec), packages/hexfield/rust/src/ (search sessions, eval cache), and apps/showcase/server/showcase/ plus apps/showcase/Dockerfile.xpu (the production deployment).

01the rules engine

The engine is a Rust crate behind a PyO3 bridge (hexo_engine._rust): Python, the search, and this website all hold opaque Rust state handles, and every placement goes through the same authority — an illegal move is a raised error, not a corrupted board. Because the board is unbounded, storage is sparse: a hash map of occupied cells plus an insertion-ordered list, no grid anywhere.

The interesting part is what a placement updates. A stone sits inside exactly 18 six-cell windows — the three line axes of the hex grid, times the six positions the stone can occupy inside a length-6 window — and those 18 are the only windows whose status that stone can change. The engine keeps a window store: for every window ever touched, two 6-bit masks (one per colour). A placement ORs one bit into 18 masks and reads the result: a window with all six bits of one colour is a win; a single-colour window with four or more stones is a threat. That is the entire win/threat detector — constant work per move, whether the game has 10 stones or 300, and never a board rescan.

Two more stores ride the same incremental discipline. The legal-move store maintains the radius-8 disk around the stones (every placement opens up to 217 new cells and closes its own), so “list the legal moves” is a copy, not a computation. And a live-threats index mirrors the window store so that “does anyone have a threat right now?” is an O(1) read — that is the question threat-space search asks before every single move, and it is why TSS can afford to ask it. When you see the search force-include a double-threat cell, the data it read was these same masks. Every mutation also returns an undo delta — the search rolls positions back instead of copying them.

The last piece is the currency the whole system trades in: the packed action id. A coordinate (q, r) becomes one u32 — each axis offset into an unsigned 16-bit lane, then q << 16 | r — chosen so plain integer order equals deterministic (q, r) order. The same encoding is implemented three times, byte-for-byte compatibly — Rust, Python, and the board JS on this site — because these ids are persisted in training shards and game records; a cross-language test pins them together.

02the game record

Every finished game — a self-play game on the training box, an arena game in the evaluator, and the game you just played on this site — is written in one binary format: .hxr. A file starts with the magic HEXOREC1 and a small header (schema version, engine rules version, backend string, the player list), then zero or more length-prefixed game payloads. Each payload is self-contained: game id, optional seed (zigzag varint), a status byte, the move stream as raw u32 little-endian packed action ids, then optional winner, placement count, and an abort record for games that died mid-run. Writers flush per game, so a crashed self-play run leaves a readable prefix rather than a corrupt file; readers reject unknown schema versions and trailing bytes loudly.

The move stream is the whole cost: 4 bytes per ply, plus roughly 60 bytes of per-game metadata. A typical ~60-ply game is about 300 bytes — the packed-id encoding means a record stores decisions, not boards, and any consumer replays them through the engine to reconstruct every position. That is literally how this site works: your game is finalized as a one-game .hxr blob into the games database, and the analysis view replays those action ids through the same Rust engine before the network ever sees the position.

03from your click to a move

When you play here, your click becomes POST /api/game/{id}/move against a FastAPI process that owns no model at all — it checks your session cookie and a token bucket, applies your stone through the Rust engine, and enqueues a bot-turn job. Jobs route sticky: game_key % workers pins each game to one of two worker processes, because that worker is where your game’s state lives — each worker holds the full checkpoint catalogue, one network and one Rust search session per checkpoint, resident for its lifetime.

Inside the worker, the session searches with the run’s as-trained knobs (Gumbel root, sequential halving, TSS — only the visit budget you picked varies), and two reuse layers do the heavy lifting. The per-game tree: the session keeps one search tree per game_key; after it plays a move it advances the root into the chosen child, so your next move usually starts from an already-grown subtree instead of scratch. The eval cache: every evaluated position is keyed by a state hash and kept in a FIFO-bounded map (the crate default holds 262,144 states; this site configures 65,536 per checkpoint session), so transpositions and re-searches never hit the GPU twice. Leaves that do miss are batched — the search collects up to 32 of them and makes one network forward per batch. The bot plays its whole turn (one or two stones) in a single job, then the tree for a finished game is explicitly discarded on its sticky worker.

The rate limits guard exactly one scarce thing: search compute. Moves are capped at 120/minute and new games at 60/hour per client, at most 16 games run concurrently server-wide and 4 per IP, and analysis requests (which can trigger small searches, capped at 64 visits) get 40/minute. Identity for all of this is the Cloudflare-reported client IP — the container has no directly exposed ingress.

04the gpu it runs on

The whole site is served by one Intel Arc A310 — a roughly $100, low-profile card — through torch 2.12.1+xpu, the PyTorch wheel line where Intel-GPU support is native (torch.xpu, no extension package). There is a wrinkle worth being honest about: the training run’s fast Triton kernels are CUDA-only and gated on x.is_cuda, so on this box every forward takes the plain eager fp32 paths — the same math, minus the hand-tuned kernels. Even so, the measured result on this deployment is 2.2× single-move speed and 2.8× concurrent throughput over the host’s 7-core CPU baseline — batching favours the GPU, which is exactly the shape of the serving load.

Correctness is not assumed. Eager-on-XPU is an untested-in-CI combination, so at startup each worker forwards one fixed position on CPU and on the XPU and compares value and policy logits. Tolerance is 10−2; the measured divergence here is about 5×10−6. If the check ever failed — mismatch, non-finite output, or the device forward raising — the worker falls back to CPU and serves slow moves rather than wrong ones.

The one operational quirk is the JIT. The first forwards after a worker starts compile the SYCL kernels for the network — about 27 seconds, once per worker (two workers, compiled independently), absorbed at startup by a deliberate warmup so no player’s first move eats it. The container runs a read-only filesystem with tunnel-only ingress, so the kernel cache lives on tmpfs: a rebooted container pays the 27 seconds once, then serves warm indefinitely.

That is the theme of the whole stack: the model is the only exotic part. Around it — incremental masks instead of rescans, four bytes a move, sticky processes, a parity check before trusting a backend — the engineering stays small enough to read in an afternoon, which is exactly what this page just did.
