The serving — how one GPU feeds 256 searches

A search is only as fast as its evaluator. hexfield_eq runs 256 games at once, each burning 256 visits per decision, against a single GPU — and the network reads a variable-size support set, so every leaf arrives as a different-shaped tensor. The serve pipeline turns that stream of small ragged positions into a few large fixed-shape forwards: one wire between the Rust engine and the Python evaluator, asynchronous submission so the search never waits, padding groups with a bounded waste budget, and pre-captured GPU graphs over cached dense weights.

01the queue and the wire

The search page ended at the continuous scheduler: 256 games drop pending leaves into one shared queue, and at roughly 1,024 pending — or earlier, when nothing can progress without results — the batch flushes to the evaluator. Duplicate positions are evaluated once per batch and cached across the run in a 262,144-state eval cache; finished games are replaced immediately, so the queue never starves while others finish.

Each flush is one payload across the Rust–Python boundary. Rust featurizes the leaves in parallel and ships the feature rows for all positions concatenated in a single f16 buffer, sorted largest-position-first, together with coordinates, neighbour tables, ray lengths, and one legal count per row. Python replies with values, priors softmaxed over each row’s legal prefix, the raw logits, and moves-left when the request asks for it.

02batching without blocking

Why asynchronous. Submission and result-draining are separate calls. The engine submits a payload — the GPU work enqueues with no sync — and immediately goes back to selecting the next batch of leaves while the forward runs; a single device sync drains the result later. The GPU and the tree search overlap instead of taking turns, which is the whole trick of feeding 256 searches from one device.

Why fixed shapes. Every position has its own node count, and a GPU hates a thousand differently-shaped small launches. So the evaluator packs rows into padding groups: each group shares one padded node count Npad, quantized to a multiple of 64, and rows are padded up to it. The packing is bounded two ways — a padding-waste cap of 18% (waste = 1 − Σn / (B·Npad)) and a B·Npad² ceiling on per-group attention cost. That buys bounded wasted math in exchange for few, fixed-shape launches — and fixed shapes are exactly what makes the graph replay of §03 possible. Sorting the wire payload largest-first is what lets these groups be cut with minimal waste.

Concretely: the walkthrough position has N = 402 nodes and rides in a group padded to Npad = 448, giving batch tensors (B, 448, 46), (B, 448, 6), (B, 448, 2), (B, 448, 12) and a (B, 448) mask.

In the figure, positions of 150–450 nodes arrive continuously on the left; a flush snapshots the pending set, sorts it largest-first, and cuts it greedily into groups — a group grows until admitting the next (smaller) row would push its waste over 18% or its B·Npad² over the ceiling. Each group then fires to the GPU while the left side keeps filling: submit is an enqueue that returns immediately, and the drain (one device sync) happens whole groups later. The sizes and rates are scaled for the figure; the quantum, the waste cap, and the existence of the B·Npad² ceiling are the production rules.

03fixed shapes, cached weights

Once groups arrive in a handful of fixed shapes, the forward itself can be frozen. Whole-group forwards are replayed as pre-captured GPU graphs keyed on the bucketed (B, Npad) shape: capture once per bucket, then replay — few, fixed-shape launches instead of re-dispatching every kernel on every flush. The computation runs in fp16, with the value and moves-left tops kept in fp32.

The tied model costs nothing extra here. The net stores only free weight blocks (84 per conv, 12 per 1×1) and materializes dense weights by index-gather; at serve time the materialized weights are cached against the parameters’ version and regenerate only when a new checkpoint loads. The head-coset channel permutations that pin one attention head per win axis are folded into those cached projection weights — so inference performs zero runtime permutes.

That closes the loop this page opened: the queue keeps the wire full, the wire keeps the groups full, the groups keep the graphs warm — and the search never stops to watch any of it happen. What the evaluated positions are for — the rows they become and the losses that train on them — is the next page.
