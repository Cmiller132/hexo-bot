The network — a graph net for an unbounded board

The board is unbounded and the playable region moves with the game, so the input has no fixed size. The network is a set network over a graph, and its input size changes every move. This page describes the architecture as implemented: the support set, the 15 per-cell features, hex convolution as message passing, global attention with a learned hex-geometry bias, and the output heads. The board figures run on real positions from the run’s snapshot data (the support and convolution figures use a quiet middlegame, 15 stones, player 0 to move; the feature inspector adds three tactical positions) with the formulas from the featurizer. The attention visualizer shows the ep70 checkpoint’s learned weights; the bias-kernel figure shows the same checkpoint’s tables.

Ground truth. Everything on this page is checked against the code: model.py, support.py, features.py, constants.py, geometry.py in packages/shrimp/python/shrimp/, and the shipped recipe configs/shrimp_main_7.toml (192 channels, 3 attention heads, trunk CCACCACCACCACCA, support radius 4; 8,128,812 parameters).

01The board as a graph

Legality reaches hex-distance 8 from the nearest stone, so the playable region grows throughout the game and can move arbitrarily far from the origin. A fixed H×W input window would waste computation on empty cells or cut off stones. Shrimp builds a support set, the finite set of cells relevant to the current position, and feeds the network that set as a variable-length list of nodes plus a neighbour graph.

The support has three layers, built in one multi-source BFS from the stones (support.py):

  • Stones: every occupied cell.
  • Legal cells: empty cells within hex-distance R of some stone. The engine's legality radius is 8; the model's radius is set by SHRIMP_SUPPORT_RADIUS, and the shipped weights use 4. The tighter radius restricts candidate moves to a smaller ring around the stones and roughly quarters the node count per evaluation.
  • Halo: the shell at distance exactly R+1, a one-cell border that gives edge cells a complete convolution stencil (§03). Halo cells are never playable.

The nodes are stored in the order [ legal | stones | halo ], each segment ascending by packed cell id. Because legal cells come first, the policy head's first legal_count outputs are exactly the legal moves (the legal-prefix invariant). There is no separate move mask that could drift out of sync with the rules. At the empty opening position the entire support is 7 nodes (the forced origin move plus its 6 halo neighbours); a crowded midgame runs to a few hundred.

02Per-cell features

Each support node carries 15 features (constants.py indices F_*, computed in features.py). All "own/opp" features are relative to the player to move. In index order:

  • 0–2 own-stone / opp-stone / empty: one-hot occupancy; empty = 1 − own − opp.
  • 3 legal: 1 on the legal prefix, 0 elsewhere.
  • 4 phase-is-second-stone: 1 on every node when the mover is placing the second stone of the turn.
  • 5 first-stone: marks where this turn's first stone landed (only during that second-stone phase).
  • 6 player-colour: 1 everywhere when player 0 is to move.
  • 7–8 own / opp recency: 1/(1+age) where age = placements_made − placement_index: the newest stone reads 1.0, the one before ½, then ⅓.
  • 9–10 opp-hot / own-hot: empty cells inside a six-cell single-colour window already holding ≥ 4 stones (threats; active once ≥ 7 stones are placed).
  • 11 distance-to-nearest-stone: the BFS distance scaled by 1/8 (the scale stays pinned to the engine radius even when the support radius is 4, so a radius-4 halo cell reads 5/8 = 0.625).
  • 12 opp-last-turn: the one or two cells of the opponent's most recent completed turn.
  • 13–14 opp-win-now / own-win-now: empty cells that complete five-in-a-window, a standing win-in-one.

The hot and win planes come from scanning every length-6 single-colour window on the three win axes, the same threat definition the search's tactical layer uses. The inspector below computes all 15 values with the real formulas, on four positions from the run's snapshot data (data/features.json): a quiet middlegame where every threat plane is zero, a single split four (three opponent-hot cells), a double threat (six), and a mid-turn position where both sides hold a standing win-in-1, which also sets the phase-second and first-stone planes. Each plane the page claims non-zero is verified: the in-page mirror of features.py recomputes the full feature matrix and checks it against a per-plane digest exported from the Python featurizer.

03Hex convolution as message passing

The trunk's local operator, HexNodeConv, is a graph message-passing step. The support construction precomputes a neighbour table: for each node, the row indices of its 6 hex neighbours in the fixed direction order D (the rotate-60 orbit of (1,0)). The forward pass gathers each node's 7 taps (itself + 6 neighbours) into a (N, 7·C) matrix and applies one GEMM against a (7, C_in, C_out) weight. The weights are direction-typed: the neighbour in the Q direction is mixed with different learned weights than the neighbour in the R direction, because the axis a line runs along determines its tactical meaning.

At the support's edge, a missing neighbour is stored as −1 by the featurizer, and the batching layer maps it to index Npad, a row of an appended all-zero vector. The gather then reads zeros for that tap, giving exact zero-padding semantics, with the output re-multiplied by the node mask so padded rows stay inert. The halo guarantees that every legal cell and stone has all 6 neighbours inside the support; only halo cells themselves have zero-taps facing outward.

One trunk C block is two of these convolutions in a post-activation residual: ReLU(x + γ · LN(conv(ReLU(LN(conv(x)))))), with a per-channel LayerScale γ initialised at 10⁻⁴ so residual branches start nearly closed. LayerNorm normalises each node independently, so statistics do not depend on how many nodes a board has. Convolution is local: information moves one ring per conv, two rings per block. A winning window is 6 cells long, boards run 20+ cells across, and the shipped trunk places two conv blocks (four convs) between attention blocks, so convolution alone cannot relate two distant threats. Attention covers that (§04). The stepper below shows the conv reach on the support graph.

04Global attention over an irregular set

The trunk's A block is full self-attention over a joint sequence of [ 8 summary tokens ; all support cells ]. Every cell attends to every other cell in both directions in a single step; the only excluded keys are batch padding, masked additively at −3·10⁴. A threat forming on the far side of the board is one attention step away. At the shipped width the attention runs 3 heads of dimension 64 (192/3; d=64 is an efficient size for GPU attention kernels), inside a standard pre-norm transformer block with a GELU MLP at hidden ratio 2.

Position enters through a learned relative-position bias; the features carry no coordinates. Each attention block has its own (237, heads) table, zero-initialised, indexed per query–key pair by the axial offset between the two cells (geometry.py rel_bias_index):

  • rows 0–216: one exact row per offset with hex-distance ≤ 8 (the 217-offset disk);
  • rows 217–224 vs 225–232: ring buckets for distance 9–16, split by whether the offset is collinear with a win axis (collinear offsets are the ones that can share a winning line);
  • row 233: everything beyond distance 16;
  • rows 234–236: cell↔token and token↔token pairs.

The bias is additive: it adjusts scores by distance and direction and never excludes a pair. Each of the five attention blocks learns its own table, and after 70 epochs the tables encode definite geometric priors. The figure below paints one block/head's table over the offset disk: the number shown for an offset is added to the attention score of every key cell at that offset from the query. Two observed patterns in the ep70 tables: block 4 head 1 suppresses the query's own cell and its entire first ring (rows −4.0 and ≈−2.6 mean) while giving its largest positive bias to offsets exactly six steps away along a win axis (mean +2.0, against +0.5 for off-axis offsets at the same distance) — a winning window is six cells long. Block 4 head 2 is close to the complement: positive on near on-axis offsets (distance-4 on-axis mean +0.8 vs +0.2 off-axis) and negative in the far rings.

The 8 summary tokens are learned vectors with no board position; they ride in the same sequence through all five blocks and serve as whole-board aggregation slots that every cell can attend to and from. Cells also attend to each other directly. The scalar heads later read specific token pairs (§05).

The visualizer below shows real attention rows from the run’s ep70 checkpoint, exported for four positions from its own games (data/attention.json). Three observed behaviours: head 0 in blocks 0 and 4 is a token sink, placing nearly all of its mass on the summary tokens for most queries (token 4 in the first block at weights ≈1.0; the last block spreads its token mass, with token 6 heaviest at 0.57 from the split-four position's last-stone query). In the split-four position, block 2's head 1 queried from the gap puts 0.89 of its mass on the stone that completes the four. In the double-threat position, block 3's head 1 queried from a threat-window cell puts 0.89 on (−3, 3), the forced blocking gap.

05The outputs

After the last attention block, one shared LN_final normalises the joint sequence and splits it back into cells and tokens. Two kinds of heads read the result. Per-cell heads (each its own HexNodeConv + ReLU + Linear(192,·)) score cells; scalar heads read a dedicated pair of summary tokens concatenated with a masked mean-pool of all cell vectors, through a Linear(576→192) reduction. Value reads tokens {0,1}, the short-term heads tokens {2,3}, moves-left tokens {4,5}; tokens 6–7 have no head attached. The heads:

  • Policy: one logit per cell; the softmax is taken over the legal prefix only, so illegal moves are unrepresentable. This is the search's prior at play time.
  • Value: distributional; 65 logits over bins spanning −1…+1, with the scalar taken as the softmax expectation (decode_binned_value). This is the leaf evaluation the search backs up.
  • Short-term value: the same 65-bin form at horizons of 2, 6 and 16 turns, a denser training signal than the final result alone.
  • Moves-left: 65 bins decoded to 0…209 moves remaining; used to prefer decisive finishes when already winning, and shown in the showcase analysis panel.
  • Train-only heads, computed in the training forward but skipped at serve time: opponent-policy (predict the opponent's next reply), an auxiliary soft policy, and a per-cell Q head (a 65-bin value estimate for every legal move). They exist to improve the trunk's representation.

The serve-time forward (forward_policy_value) computes only policy + value, plus moves-left on request; the auxiliary heads add no cost at play time.
